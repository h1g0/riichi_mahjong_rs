# CLAUDE.md

## Project Overview

Japanese Riichi Mahjong game implemented in Rust. Runs as a native desktop app and in the browser via WASM.

## Crate Structure

- **`mahjong-core`** — Pure game logic: tile representation, shanten calculation, yaku evaluation, fu/score calculation. No UI.
- **`mahjong-server`** — Game progression and rule handling for local matches. Includes CPU opponent AI.
- **`mahjong-client`** — Macroquad-based GUI client. Supports both native and WASM builds.

## Common Commands

```sh
# Run tests
cargo test
cargo test -p mahjong-core
cargo test -p mahjong-server

# Run native client
cargo run -p mahjong-client

# Build WASM client
cargo build -p mahjong-client --target wasm32-unknown-unknown --release

# Serve WASM locally (after building)
python -m http.server 8080
# Then open index.html in a browser
```

## WASM Build Notes

- Requires `wasm32-unknown-unknown` target: `rustup target add wasm32-unknown-unknown`
- `.cargo/config.toml` sets `getrandom_backend="custom"` for WASM — this is intentional to avoid wasm-bindgen
- Do not introduce wasm-bindgen dependencies

## Key Directories

- `crates/` — workspace crates
- `assets/fonts/` — NotoSansJP-Regular.ttf (loaded at runtime)
- `assets/images/` — tile PNGs and score stick PNGs
- `public/` — web deployment output (generated by build scripts)
- `scripts/` — Vercel build/install scripts
- `index.html` — local WASM entry point

## Architecture Notes

- `mahjong-client/src/adapter.rs` — bridges the client UI with the server logic
- `mahjong-server/src/round.rs` — largest module; handles full turn sequence (~77KB)
- `mahjong-server/src/cpu/` — CPU AI with personalities, defense, and move evaluation
- Yaku checks are organized by han level under `mahjong-core/src/winning_hand/`

## Development Guidelines

### Game Logic

- Before adding game logic to `mahjong-server` or `mahjong-client`, check whether it already exists in `mahjong-core`.
- After completing an implementation, verify that the build succeeds and all unit tests pass (`cargo build && cargo test`).
- Add unit tests for any new functionality.
- Add regression tests for any bug fixes.

### Pre-Commit Checklist

Before committing, ensure there are no formatting issues or warnings:

```sh
cargo fmt
cargo clippy
```

Both commands must complete without errors or warnings.

### Coding Conventions

- Commit messages and PR descriptions must be written in English.
- Branch names follow the format `{type}/{issue-number}-{english-branch-name}`.
  - Types: `fix` (bug fix), `feat` (new feature), `misc` (other)
  - Example: `fix/#87-hide-dora-on-draw-result`

## Deployment

Vercel deployment is pre-configured via `vercel.json`. The build runs `scripts/vercel-install.sh` then `scripts/vercel-build.sh`, placing web assets under `public/`.
